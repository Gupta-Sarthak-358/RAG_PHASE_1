# CHRONOLOGICAL DESIGN DECISIONS  (Why things are the way they are)
# ═══════════════════════════════════════════════════════════════════

# ── v1 baseline decisions ─────────────────────────────────────────
#
# WHY Qwen2.5 7B GGUF via llama-cpp-python for Phase 1?
#   Better at precise JSON extraction tasks than Mistral.
#   Mistral will be used in Phase 2 for unstructured summarization tasks.
#   Fully local, no API cost, deterministic with seed=42, fits in 8GB VRAM.
#   Q4_K_M quantization gives acceptable quality with ~5GB model size.
#
# WHY 8K context window?
#   Chapters average 1500 words (~2000 tokens). Prompt + schema + response
#   lands around 4000-5000 tokens. 8K gives room for longer chapters.
#   MAX_CHAPTER_WORDS = 5000 is a safety truncation before this limit.
#
# WHY FAISS IndexFlatIP with normalized vectors?
#   Normalized inner product = cosine similarity.
#   sentence-transformers already normalizes embeddings, so this is free.
#   IndexFlatIP is exact (no approximation), fast enough for ~70 chapters.
#
# WHY retry once on extraction failure?
#   LLMs sometimes produce malformed JSON on first pass.
#   A second attempt with the same prompt often succeeds (temperature=0.1,
#   so outputs are ~deterministic but not completely frozen).
#
# WHY filter hallucinations at all?
#   Small 7B models (like Mistral or Qwen) can hallucinate character names.
#   Early test: 15-20% of extracted characters were pure inventions.
#   Simple fix: if name doesn't appear in chapter text → discard.


# ── v2 decisions: stable character IDs ───────────────────────────
#
# PROBLEM: old system keyed characters by display name.
#   "Akane" and "Akane Uzumaki" create two entries. Merging by name
#   string is fragile and breaks when names vary.
#
# WHY stable char_NNNN IDs instead of name-derived canonical_ids?
#   Name-derived IDs ("akane_uzumaki") break on name changes.
#   "Akane Yanagi/Uzumaki" can't be converted to a clean snake_case
#   identifier without collision risk.
#   char_0001 is opaque but guaranteed unique and collision-safe.
#
# WHY an ID counter file (id_counter.json)?
#   Deleted characters should not recycle their IDs.
#   Keeping a persistent counter ensures monotonic increase with no gaps.
#   Counter is recovered from existing IDs if the file is lost.
#
# WHY ambiguity detection returns (None, True) instead of auto-merging?
#   Auto-merging "Naruto" to "Naruto Uzumaki" is correct.
#   But auto-merging "Uzumaki" to any Uzumaki is wrong.
#   The ambiguous flag lets the caller create a fresh stub rather than
#   silently corrupting two different characters into one entry.
#
# WHY _MERGE_GROUPS as explicit config?
#   Some merges (Akane Yanagi / Akane Uzumaki → char_0001) cannot be
#   detected automatically without story context. These are one-time
#   manual declarations that run once during migration.
#
# Relationship format during v2: dict keyed by char_id
#   {"char_0002": {"current_status": "ally", "since_chapter": 5}}
#   NOTE: This format is superseded in v7 — see relationship repetition section.


# ── v3 decisions: relationship events, emotional deltas, threads ──
#
# WHY separate relationship_events list instead of overwriting relationships?
#   "Naruto and Sasuke fight" in ch12 doesn't erase their alliance from ch8.
#   A single relationship label (ally/enemy) loses the timeline.
#   Event-based history preserves the progression: met → ally → fight → reconcile.
#
# WHY emotional_deltas as append-only log per character per chapter?
#   Phase 1 does not need a running emotional state tracker.
#   Storing raw deltas per chapter makes it easy to replay history or
#   feed the data into a Phase 2 tracker without re-extracting.
#   Delta bounds: -0.3 to +0.3 per chapter. Large swings are narrative noise.
#
# WHY thread_progression with status_change values instead of just description?
#   Freeform descriptions don't allow queries like "all threads that escalated".
#   Enum status_change (advanced|escalated|complicated|stalled|resolved) enables
#   structured querying while justification preserves the narrative reason.
#
# WHY purge_chapter preserves character entries but removes chapter data?
#   Characters exist across the whole story. The chapter data (events, deltas,
#   thread updates, relationship events) is chapter-specific and safe to replace.
#   Removing a character entry and re-creating it would lose data from other chapters.


# ── v4 decisions: name validation and alias matching ─────────────
#
# PROBLEM: LLMs output role/group labels as character names.
#   "The Jonin Commander", "Team 7", "several shinobi" appeared as characters.
#   These are not individuals. They pollute the character database.
#
# WHY reject prefixes ("the ", "a ", "an ", "some ", "several ")?
#   Real character names never start with articles.
#   "The Jonin" is always a group reference, not a person.
#
# WHY reject role words (ninja, shinobi, jonin, chunin, genin, captain, etc.)?
#   Role descriptors as the full name = group hallucination.
#   "Jonin" with nothing else is a rank, not a name.
#   NOTE: "Captain Yamato" passes validation (Yamato is a real name).
#   The rejection is for names WHERE ALL WORDS are role words.
#
# WHY reject commas and "and"?
#   "Naruto and Sasuke" is the LLM cramming two characters into one slot.
#   Any name with a comma or " and " is a multi-person entry.
#
# WHY NAME_MAX_WORDS = 3?
#   Real names: 1-3 words (given name + family name + optional alternate).
#   "The fake Konoha shinobi leader" is clearly a description, not a name.
#
# WHY honorific stripping in the MERGER not the EXTRACTOR?
#   The extractor keeps names as they appear in the text (for faithfulness).
#   The merger strips honorifics (-sama, -kun, -chan, -hime, etc.) to improve
#   the alias lookups. "Tsunade-Hime" → try matching "Tsunade" after strip.
#
# WHY _is_valid_character_name returns (bool, reason) instead of just bool?
#   Structured rejection reason makes console output specific and debuggable.
#   "✗ rejected: 'The Jonin' (starts with 'the')" is far more useful than
#   a bare rejection with no explanation.


# ── v5 decisions: evidence schema and JSON repair ─────────────────
#
# WHY add evidence fields to every extracted item?
#   LLMs hallucinate confidently. A character can be "extracted" from text
#   where they never appear. Requiring an exact quote as evidence anchors each
#   extraction to source text. This is the primary hallucination filter.
#
# WHY evidence as an EXACT quote, not a paraphrase?
#   Paraphrases require semantic similarity to verify — expensive.
#   Exact quotes can be verified with a simple substring check.
#   The prompt says: "copy the sentence exactly as it appears".
#
# WHY keep the name-in-text fallback when evidence is absent?
#   Some older data had no evidence fields.
#   If the LLM doesn't produce evidence, falling back to name-in-text
#   preserves the useful extraction instead of discarding it entirely.
#
# WHY sort events and relationship_events by chapter after merge?
#   Chapters can be reprocessed out of order.
#   A chapter 4 reprocess after chapter 10 was processed appends at the end.
#   Sorting ensures chronological order for any consumer of the event list.
#
# WHY the JSON repair pipeline?
#   Language models regularly produce minor JSON syntax errors:
#     - Trailing commas         → stripped by regex
#     - Adjacent strings with no comma between  → ",\n" inserted
#     - Adjacent objects with no comma  → }{ → },{
#     - Single-quoted strings   → converted to double-quoted
#     - Newlines inside strings → escaped
#   Each repair step solves a specific observed failure mode.
#   The pipeline is applied to the entire raw output, then to bracket-
#   extracted substrings, then to fenced code blocks — most lenient last.


# ── v6 decisions: simplifying evidence verification ───────────────
#
# REMOVED: _normalize_for_evidence() function
#   Reason: whitespace-normalizing both evidence and chapter text before
#   comparison added complexity with minimal benefit. The main source of
#   evidence mismatch is the LLM paraphrasing, not whitespace differences.
#   Simple case-insensitive substring check is sufficient.
#
# REMOVED: Fuzzy word-overlap matching (EVIDENCE_FUZZY_THRESHOLD)
#   Reason: If the LLM paraphrases the evidence, the quote is already wrong.
#   Fuzzy matching accepts paraphrases, which defeats the purpose of evidence.
#   Either the exact sentence is there or the extraction is suspect.
#   "~ evidence fuzzy-matched" was noise, not a useful signal.
#
# REMOVED: EVIDENCE_FUZZY_THRESHOLD constant from config.py
#   No longer needed.
#
# KEPT: EVIDENCE_MIN_WORDS = 3
#   A 1-2 word evidence quote is not meaningful ("she said" proves nothing).
#   Minimum 3 words ensures the quote is a real phrase, not a fragment.
#
# KEPT: name-in-text fallback
#   When evidence field is absent entirely, name-in-text is still applied.
#   This handles old data and cases where the LLM omits evidence.
#
# RESULT: _verify_evidence() is now 5 lines:
#   evidence.strip().lower() in chapter_text.lower()
#   with a min-word guard. No normalization, no fuzzy matching.


# ── v7 decisions: relationship repetition system ──────────────────
#
# PROBLEM: LLMs overreact to single events.
#   "Naruto and Sasuke argue" → extractor outputs: enemy
#   But they've argued a thousand times and remained allies.
#   A single event should never flip a relationship state.
#
# WHY signal counting instead of event-based instant promotion?
#   Story relationships build gradually through repeated behavior patterns.
#   One argument = a moment. Three arguments = a pattern = conflict.
#   Counting preserves the event (for history) but gates the state change.
#
# WHY the specific thresholds?
#   betrayal >= 1  → enemy (instant flip)
#     Reason: betrayal is a singular act that correctly redefines a relationship.
#     A friend who betrays you is now an enemy, regardless of prior history.
#     "Humans forgive slowly but betrayals hit hard."
#
#   kiss/confession/alliance >= 1 → romantic/ally (instant)
#     Reason: These are singular declarative events. A confession is not a
#     pattern — it's a pivot point. An explicit alliance agreement is a fact.
#
#   romantic_progression >= 2 → romantic
#     Reason: ambiguous romantic tension requires repetition to confirm.
#     One scene of "romantic tension" could be reader projection.
#
#   cooperation >= 2 → ally
#     Reason: one cooperative act could be circumstantial. Two signals
#     indicate a deliberate pattern of working together.
#
#   trust_gain >= 2 → friend
#     Reason: trust is built over multiple demonstrated moments.
#
#   argument >= 3 → conflict
#     Reason: the highest bar because arguments are common and rarely
#     define a permanent relationship shift in long-running stories.
#
# WHY REL_MIN_INTERACTIONS = 2?
#   Prevents "met + argument = enemy" in a single chapter.
#   Characters who just met cannot have a defined relationship state yet.
#   The first chapter of any pair is introductory, not definitional.
#
# WHY does _set_static_relationship bypass thresholds?
#   Explicit declarations in the extraction ("she is his sister") are
#   authoritative facts, not signals to count. Family, mentor, and other
#   explicitly-stated labels are not patterns — they are story facts.
#   _record_rel_event is for dynamic events. _set_static_relationship
#   is for ground-truth declarations.
#
# NEW relationship format:
#   {
#     "state": "neutral",          ← derived from signal thresholds
#     "history": [                  ← every event, in order
#       {"chapter": 4, "event": "argument"},
#       {"chapter": 5, "event": "argument"},
#       {"chapter": 6, "event": "argument"}
#     ],
#     "signals": {                  ← counts per event_type
#       "argument": 3
#     }
#   }
#
# Migration: old {current_status, since_chapter} format is migrated on-the-fly
#   in _ensure_rel_entry() when a character's relationship entry is first accessed.


# ── v8 decisions: minor fixes & optimizations ──────────────────────────
#
# WHY reintroduce fuzzy matching?
#   Exact match was too brittle due to minor differences in character names
#   or light paraphrasing. Reintroduced with a lower threshold (0.6) to allow
#   some leniency while still anchoring to the text.
#   Added EVIDENCE_FUZZY_THRESHOLD = 0.6.
#
# WHY normalize whitespace in evidence matching?
#   Line breaks in dialogue broke exact matching. Normalizing whitespace
#   (converting all newlines/tabs/spaces to a single space) fixes this.
#
# WHY shrink the prompt?
#   Removed the explanatory essay and reduced rules from ~40 lines to 10.
#   Schema remains intact. Saves 500-800 tokens per call, reducing latency.
#
# WHY remove word-overlap in canon merger (_find_character_id)?
#   Pass 3 (significant-word overlap) caused phantom merges (e.g., "The" matching
#   "The Jonin" matching "The fake Konoha ninjas"). Removed to enforce stricter
#   matching (only exact and first-name match).
#
# WHY skip retry on success?
#   No need to loop if the first attempt parses successfully. Returns immediately.
#
# WHY lowered max tokens and GPU offloading?
#   Added MODEL_GPU_LAYERS = -1 in config to enable full GPU offload via llama.cpp.
#   Reduced MODEL_MAX_TOKENS = 1200 since extractions are typically shorter.
#   Reduces extraction time from ~60-90s to ~25-45s per chapter.


# ── v9 decisions: normalization layer ──────────────────────────────────────────
#
# PROBLEM: Raw extractions contain highly varied labels and redundant entries.
#   "Fireball Jutsu", "Fireball Technique", "Katon: Fireball" are extracted as
#   separate abilities. Minor variations in plot thread names or event descriptions
#   cause duplicate entries in the JSON.
#
# WHY a dedicated normalizer.py?
#   Keeps canon_merger.py clean and focused on merging logic. The normalizer
#   acts as an intermediate pure Python layer that deduplicates and sanitizes
#   the extraction dictionary *before* it touches the story state.
#
# WHY RapidFuzz?
#   Much faster and more accurate than difflib for string comparison. It handles
#   the fuzzy matching required for deduplicating plot threads, abilities, and events.
#   A fallback to difflib is provided if rapidfuzz is uninstalled.
#
# WHY ABILITY_NOISE_WORDS = {"technique", "jutsu", "no"}?
#   Removing these common suffixes before comparing two abilities dramatically
#   improves the accuracy of deduplication (e.g., "Fireball Jutsu" vs "Fireball").
#
# WHY REL_PRIORITY overriding?
#   Relationships like "family" are permanent. A sequence of "allies" -> "family"
#   is a valid upgrade, but "family" -> "allies" is a downgrade and should be
#   ignored. The new config priority scale ensures explicit static relationships
#   only move upwards in significance.

# ═══════════════════════════════════════════════════════════════════
# FILTER PIPELINE  (order matters)
# ═══════════════════════════════════════════════════════════════════

# For each extracted item:
#
# 1. NAME VALIDATION (_is_valid_character_name)
#    Rejects: empty, >3 words, commas, "and", reject prefixes (the/a/an...),
#             role/group words (ninja/shinobi/jonin/chunin/genin/captain...),
#             all-lowercase multi-word (descriptions), pure role names (jonin alone)
#
# 2. EVIDENCE VERIFICATION (_verify_evidence)
#    If evidence present:
#      - Must be >= EVIDENCE_MIN_WORDS (3) long
#      - Must appear as exact substring in chapter text (case-insensitive)
#      - If passes → accept, stop here
#      - If fails → reject (print ✗ reason)
#    If evidence absent → fall through to step 3
#
# 3. NAME-IN-TEXT FALLBACK (_name_in_text)
#    Check if name (or any substantial part) appears in chapter text.
#    If yes → accept with warning ("~ no evidence, accepted via name-in-text")
#    If no  → reject ("✗ no evidence, name not in text")
#
# 4. NORMALIZATION LAYER (normalizer.py)
#    Applied to the entire extraction dictionary before merging:
#      - Cleans character names
#      - Fuzzy deduplicates abilities, major events, relationship events
#      - Matches new plot threads against existing unresolved threads
#
# Special cases:
#   major_events with no evidence → kept (events are descriptions, no name to check)
#   thread_progression → never filtered (thread names are abstract labels)


# ═══════════════════════════════════════════════════════════════════
# RELATIONSHIP STATE DERIVATION  (v7)
# ═══════════════════════════════════════════════════════════════════

# Two paths into the relationship system:
#
# PATH A: _set_static_relationship(chars, from_cid, to_cid, rel_type, chapter)
#   Used for: explicitly-stated relationships ("she is his sister")
#   Behavior: writes state directly without going through thresholds
#   Examples: family, mentor, student, romantic (when explicitly declared)
#
# PATH B: _record_rel_event(chars, from_cid, to_cid, event_type, chapter)
#   Used for: dynamic events from relationship_events extraction
#   Behavior: appends to history, increments signal count, calls _compute_rel_state
#   Both directions are updated simultaneously (from→to AND to→from)
#
# _compute_rel_state(signals, history) → str:
#   Guard: if len(history) < REL_MIN_INTERACTIONS (2) → "neutral"
#   Then checks REL_SIGNAL_THRESHOLDS in priority order:
#     betrayal >= 1          → "enemy"
#     romantic_progression>=2 → "romantic"
#     kiss >= 1              → "romantic"
#     confession >= 1        → "romantic"
#     cooperation >= 2       → "ally"
#     alliance >= 1          → "ally"
#     trust_gain >= 2        → "friend"
#     argument >= 3          → "conflict"
#     trust_loss >= 3        → "rival"
#   Default → "neutral"
#
# The order of thresholds matters (betrayal is checked first, highest priority).


# ═══════════════════════════════════════════════════════════════════
# CHARACTER IDENTITY SYSTEM  (v2, unchanged)
# ═══════════════════════════════════════════════════════════════════

# Lookup pipeline (_find_character_id):
#   Pass 1: exact normalized match against display_name and aliases
#   Pass 2: significant-word overlap (words >3 chars), best match wins
#   Ambiguity: if two characters tie → (None, True), new stub created
#
# Auto-create stub on _resolve_char_id:
#   If no match found → mint new char_NNNN, initialize empty entry
#   If ambiguous → warn + mint new separate entry
#
# Honorific stripping during lookup:
#   Strips: -sama, -kun, -chan, -hime, -dono, -sensei, -senpai
#   Strips title prefixes: Lord, Lady, Prince, Princess, Sir
#   Original lookup tried first, then stripped variant
#
# _MERGE_GROUPS:
#   Explicit one-time migration declarations for aliases the system
#   cannot detect automatically (e.g., Akane Yanagi AND Akane Uzumaki
#   are both the same person — the system cannot infer this from names).
