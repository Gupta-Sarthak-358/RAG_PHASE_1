> A local-first, LLM-powered narrative co-writing system.
> Phase 1 extracts and indexes story canon. Phase 2 uses that canon to plan, expand, and validate future chapters.

## 1. Project Overview

The Story Canon Engine is a **two-phase narrative AI system** designed to help authors maintain consistency and narrative momentum across long-form stories (100+ chapters).

### Phase 1 — Canon Extraction
- Reads your `.txt` chapter files
- Uses a local LLM (Qwen 2.5 7B) to extract characters, events, abilities, and plot threads into structured JSON
- Merges and deduplicates all data into `output/story_state.json`
- Builds a FAISS semantic search index over all chapters for retrieval

### Phase 2 — Narrative Co-Writer
- Reads Phase 1's `story_state.json` to understand the current world state
- Tracks **thread health** (dormancy, pressure, escalation), **tension arc**, and **character emotions**
- Generates a scene-by-scene **outline** for the next chapter using the LLM + RAG retrieval
- Expands each scene into full prose
- **Validates** the expanded chapter against the outline (report only — never auto-corrects)
- Provides **forecasting**, **metrics dashboards**, and narrative telemetry

### How They Interact

```
Phase 1                          Phase 2
────────────────────────────     ────────────────────────────────────────
chapters/*.txt                   thread_health_state.json
       │                         tension_state.json
       ▼                         emotional_state.json
  extractor.py ──► story_state.json ──► outline_planner.py ──► expander.py
  embedder.py  ──► index/             narrative_metrics.py      validator.py
                                      forecasting.py
```

Phase 2 **always reads from Phase 1's output**. The `update` command is gated so Phase 2 state cannot advance past Phase 1 canonisation.

## 2. Architecture

### Module Dependency Graph

```
phase1:
  config ◄── all modules
  model_loader ◄── extractor, summarizer, outline_planner, expander, validator
  file_processor ──► extractor ──► normalizer ──► canon/merger/ ──► story_state.json
  embedder ──► retriever ──► outline_planner, expander

phase2:
  phase2_config ◄── all Phase 2 modules
  story_state.json ◄── thread_health, emotional_state, validator
  thread_health ─┐
  tension_model ─┼──► outline_planner ──► expander ──► validator
  emotional_state┘         │
                     narrative_metrics
                     forecasting
```

### Data Flow (One Full Chapter Cycle)

```
direction text (author input)
       │
       ▼
outline_planner.py    ← tension_model bounds
       │               ← thread_health pressure
       │               ← emotional_state baselines
       │               ← retriever RAG context
       ▼
chapter_N_outline.json   ← human reviews / edits
       │
       ▼
expander.py (scene by scene)   ← RAG context per scene
       │
       ▼
chapter_N_draft.txt          ← human reviews / edits
       │
       ▼
validator.py                 ← LLM + deterministic checks
       │
       ▼
chapter_N_report.json        ← human reviews report
       │
       ▼
[human saves final text to chapters/]
       │
       ▼
main.py extract              ← Phase 1 canonises new chapter
       │
       ▼
phase2_main.py update N      ← Phase 2 state advances
```

## 4. Setup

### Requirements
- Python **3.10+**
- A GPU with CUDA support strongly recommended (CPU inference is very slow)
- ~8GB disk space for the model

### 1. Clone or download the repository

### 2. Create a virtual environment

**Linux / WSL (recommended):**
```bash
python3 -m venv venv
source venv/bin/activate
```

**Windows PowerShell:**
```powershell
python -m venv venv
.\venv\Scripts\activate
```

### 3. Install dependencies

**With GPU (CUDA):**
```bash
CMAKE_ARGS="-DGGML_CUDA=on" pip install llama-cpp-python --force-reinstall --no-cache-dir
pip install -r requirements.txt
```

**CPU only:**
```bash
pip install -r requirements.txt
```

### 4. Download a model

Place a `.gguf` model in the `models/` folder. The system defaults to:
```
models/qwen2.5-7b-instruct-q4_k_m.gguf   ← Phase 1 (strict JSON extraction)
```
Update the path in `src/config.py` if you use a different model.

### 5. Add your chapters

Place your chapter `.txt` files in the `chapters/` folder. Each file name must contain the chapter number, e.g. `Chapter 42 - Title.txt`.

## 5. Quick Start

```bash
cd RAG_PHASE_1

# Option A — Interactive menu (recommended)
python run.py

# Option B — Direct CLI

# Phase 1: Extract all new chapters
python src/main.py extract

# Phase 2: Initialise from Phase 1 data (run once)
python src/phase2_main.py init

# Phase 2: Generate an outline for the next chapter
python src/phase2_main.py outline 69 "Akane discovers the sealed gate"

# Phase 2: Expand the outline to a full draft
python src/phase2_main.py expand 69

# Phase 2: Validate the draft against the outline
python src/phase2_main.py validate 69
```

## 6. Full Workflow Example

Here is the complete cycle for writing chapter 69:

```bash
# ── Step 0: Confirm Phase 1 is up to date ──────────────
python src/main.py extract          # canonise any new chapters first

# ── Step 1: Check narrative state ──────────────────────
python src/phase2_main.py dashboard   # tension, top threads, active characters
python src/phase2_main.py forecast    # 3-chapter pressure projection

# ── Step 2: Generate the outline ───────────────────────
python src/phase2_main.py outline 69 "Akane discovers the sealed gate"
# outputs:  output/outlines/chapter_069_outline.json
# ↑ Edit the JSON directly if you want to change anything

# ── Step 3: Approve the outline ────────────────────────
python src/phase2_main.py approve 69
# marks it 'approved' so the expander can proceed

# ── Step 4: Expand to full prose ───────────────────────
python src/phase2_main.py expand 69
# outputs:  output/drafts/chapter_069_draft.txt
# ↑ Edit the draft before saving

# ── Step 5: Validate the draft ─────────────────────────
python src/phase2_main.py validate 69
# outputs:  output/validations/chapter_069_report.json
# Shows: outline adherence · emotional compliance · thread progression · continuity

# ── Step 6: Finalize and save ──────────────────────────
# Copy output/drafts/chapter_069_draft.txt → chapters/chapter_069.txt
# Make any final edits

# ── Step 7: Canonise the new chapter ───────────────────
python src/main.py extract

# ── Step 8: Advance Phase 2 state ──────────────────────
python src/phase2_main.py update 69
# Updates thread health, tension, and emotional state for ch 69
```

## 7. Phase 1 Command Reference

All Phase 1 commands are run via `python src/main.py <command>`.

| Command | Description |
|---|---|
| `extract` | Extract canon from all new unprocessed chapter files |
| `reprocess [N]` | Re-extract the last N chapters (default: 5) |
| `build_index` | Build the FAISS semantic search index |
| `summary` | Generate `Canon_Bible_Summary.txt` |
| `query "<text>"` | Semantic search across all chapter chunks |
| `migrate` | One-time migration of legacy character IDs to `char_NNNN` format |
| `normalize` | Run global character deduplication pass |
| `recap [N]` | Generate a story recap focusing on the last N chapters (default: 10) |
| `snapshot` | Print a quick world state summary (active threads, key figures) |

---

## 8. Phase 2 Command Reference

All Phase 2 commands are run via `python src/phase2_main.py <command>`.

### Core Workflow

| Command | Description |
|---|---|
| `init` | Initialise Phase 2 state from existing Phase 1 data. Run once. |
| `outline <ch> "<direction>"` | Generate a scene-by-scene outline for chapter `<ch>` |
| `approve <ch>` | Mark the outline as approved for expansion |
| `expand <ch>` | Expand the approved outline into full prose draft |
| `validate <ch>` | Validate the draft against the outline (report only) |
| `update <ch>` | Update Phase 2 state after Phase 1 has canonised the chapter |

### Telemetry

| Command | Description |
|---|---|
| `dashboard` | One-glance view: arc phase, tension, top threads, top characters |
| `metrics` | Full narrative telemetry panel |
| `forecast` | 3-chapter tension + thread pressure projection |

### Debugging

| Command | Description |
|---|---|
| `inspect <target>` | Dump a state file. Targets: `threads`, `tension`, `emotional`, `story`, `conflicts` |
| `set_tension <0.0-1.0>` | Manually override the current tension value |
| `set_importance "<thread>" <1-5>` | Manually override a thread's importance level |
| `set_emotion "<char>" <dim> <0.0-1.0>` | Manually override a character's emotional dimension |

## 9. Troubleshooting

### Output files are corrupted or the system is in a bad state

All output files are crash-safe (atomic writes), but if you ever need to reset:

```bash
# Reset ALL Phase 2 state (keeps Phase 1 data)
rm output/thread_health_state.json
rm output/tension_state.json
rm output/emotional_state.json

# Re-initialise
python src/phase2_main.py init
```

```bash
# Full reset (start from scratch)
rm -rf output/
python src/main.py extract        # re-extracts everything from chapters/
python src/main.py build_index
python src/phase2_main.py init
```

### Extraction fails for a chapter

Check the raw LLM output in `logs/` if available. The extractor validates JSON schema and will skip malformed chapters. Try:

```bash
python src/main.py reprocess 1    # reprocess the last chapter only
```

### A character appears as a duplicate with a different name

1. Edit `output/aliases.json` and add the alias → canonical ID mapping.
2. Run `python src/main.py normalize` to re-deduplicate.

### Phase 2 `update` raises "Phase 1 extraction not completed"

You must run `python src/main.py extract` before running `phase2_main.py update <ch>`. This is enforced intentionally to prevent state divergence.

### The model is too slow

Enable GPU layers. Edit `src/config.py`:
```python
MODEL_GPU_LAYERS = 35  # increase this to offload more to GPU
```

Or reinstall llama-cpp-python with CUDA support:
```bash
CMAKE_ARGS="-DGGML_CUDA=on" pip install llama-cpp-python --force-reinstall --no-cache-dir
```
